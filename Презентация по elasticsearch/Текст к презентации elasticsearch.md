## Слайд 1. Обычный поиск против Elasticsearch

Всем привет. Сегодня мы расскажем, чем обычный поиск отличается от Elasticsearch.

Главная идея такая: обычный поиск чаще всего просто проверяет, есть ли нужный текст в строке. Например, пользователь написал слово `тюрьма`, и приложение ищет, встречается ли это слово в названии или описании фильма.

Elasticsearch работает иначе. Он пытается не просто найти совпадение букв, а понять, какие документы лучше всего подходят под запрос. То есть обычный поиск отвечает на вопрос: “есть ли такая строка?”, а Elasticsearch отвечает: “какой результат самый подходящий?”.

Разбирать это будем на примере приложения MovieSearch. В нём есть база фильмов, простой поиск через SQL и вариант поиска через Elasticsearch.

## Слайд 2. Маршрут рассказа

Сначала я коротко объясню, что вообще происходит в приложении.

Потом мы посмотрим на простой поиск. Там используется SQL-запрос с `LIKE`. `LIKE` — это способ сказать базе данных: “найди строки, где внутри текста встречается вот это слово или фраза”.

После этого перейдём к Elasticsearch. Там появятся новые понятия: индекс, analyzer, relevance, scoring. Я буду объяснять их простыми словами.

В конце сравним два подхода и ответим на главный вопрос: когда достаточно обычного поиска, а когда уже нужен Elasticsearch.

## Слайд 3. Демо-система

На этом слайде показана общая схема приложения.

Пользователь открывает страницу поиска и вводит запрос. Лучше брать понятные примеры из базы фильмов: `бойцовский клуб`, `тюрьма`, `заключённые` или `побег`.

Почему именно такие примеры? `бойцовский клуб` — это точная фраза из названия фильма. `тюрьма` часто встречается в описаниях. `заключённые` помогает поговорить про формы слов. `побег` показывает обычное совпадение по части названия.

Запрос приходит в Spring MVC controller. Controller передаёт его в сервис. А сервис уже решает, каким способом искать.

Есть первый путь: пойти прямо в PostgreSQL и искать по таблице `movies`.

Есть второй путь: пойти в Elasticsearch, найти подходящие документы в индексе `movies_index`, а потом по найденным id получить полные данные фильмов из PostgreSQL.

Важно: в моих локальных папках две ветки сейчас выглядят одинаково по файлам. Поэтому я сравниваю не две разные папки построчно, а два поисковых метода, которые есть в коде: простой поиск и поиск через Elasticsearch.

## Слайд 4. Обычный поиск через LIKE

Теперь посмотрим на обычный поиск.

В коде он выглядит примерно так: база берёт поле `movie`, то есть название фильма, и поле `overview`, то есть описание фильма. Потом проверяет: содержит ли одно из этих полей текст, который ввёл пользователь.

Вот здесь используется `LIKE`.

Если написать:

```sql
LIKE '%король%'
```

это значит: найди строки, где где-то внутри текста встречается слово `король`.

Проценты слева и справа означают “до этого слова может быть любой текст, и после него тоже может быть любой текст”.

После поиска результаты сортируются по рейтингу фильма. То есть простой поиск сначала спрашивает: “где есть такое слово?”, а потом сортирует найденное по `ratingBall`.

Это простой и понятный способ. Он хорошо подходит, если данных мало и требования к поиску простые.

## Слайд 5. Где LIKE подходит, а где начинает мешать

`LIKE` сам по себе не плохой. Он нормальный инструмент для простых задач.

Например, если у нас админка и нужно быстро найти запись по части названия, `LIKE` отлично подходит. Он простой, понятный, не требует отдельного сервера и отдельного индекса.

Но проблема начинается, когда пользователь ожидает поведение настоящего поиска.

Например, пользователь пишет `тюрьма`, а в описании фильма написано `тюрьме`, `тюрьму` или `тюремный`. Для человека это похожие слова. Но для простого `LIKE` это могут быть разные строки.

Ещё пример: слово найдено в названии фильма и слово найдено где-то глубоко в описании. Для пользователя совпадение в названии обычно важнее. Но простой поиск сам по себе этого не понимает.

То есть `LIKE` подходит для фильтра. Но если мы хотим именно поисковый опыт, где есть язык, смысл, важность полей и нормальный порядок результатов, тогда нужен другой инструмент.

## Слайд 6. Что добавляет Elasticsearch

Elasticsearch добавляет отдельную структуру для поиска.

PostgreSQL остаётся основной базой. Там лежит полная информация о фильмах: название, год, страна, рейтинг, описание, актёры и так далее.

Но для поиска мы создаём отдельный документ `MovieDoc`. Он проще: в нём есть `id`, `movie` и `overview`.

Дальше эти документы сохраняются в Elasticsearch в индекс `movies_index`.

Индекс можно представить как специальный справочник для быстрого поиска. Как предметный указатель в книге: слово указывает на страницы, где оно встречается.

Когда пользователь ищет, Elasticsearch возвращает документы в порядке релевантности. Потом приложение берёт id этих документов и достаёт полные фильмы из PostgreSQL.

То есть PostgreSQL хранит правду, а Elasticsearch помогает искать.

## Слайд 7. Analyzer: как Elasticsearch готовит текст

Чтобы искать лучше, Elasticsearch сначала обрабатывает текст.

Это делает analyzer. Analyzer — это набор правил, которые превращают обычный текст в удобную форму для поиска.

В проекте используется `russian_analyzer`. Он делает три важные вещи.

Первая: разбивает текст на отдельные слова. Это называется tokenizer.

Вторая: приводит слова к нижнему регистру. Например, `Король` и `король` становятся одинаковыми.

Третья: применяет русский stemmer. Stemmer пытается привести разные формы слова к общей основе.

Например, для человека слова `заключённый`, `заключённых`, `заключение` связаны по смыслу. Elasticsearch может подготовить такие слова так, чтобы поиск работал гибче, чем обычное сравнение строк.

Главная мысль: обычный поиск сравнивает сырые строки. Elasticsearch сначала готовит текст, а потом ищет по подготовленным терминам.

## Слайд 8. Релевантность и веса

Теперь самое важное: релевантность.

Релевантность — это насколько хорошо результат подходит под запрос пользователя.

В обычном поиске часто всё просто: слово есть или слова нет. А в Elasticsearch можно сказать: если слово найдено в названии фильма, это важнее. Если слово найдено только в описании, это тоже полезно, но чуть менее важно.

В коде это видно по весам:

```text
movie^4
overview^3
```

Это значит: совпадение в поле `movie` получает больший вес, чем совпадение в `overview`.

Также есть `match_phrase`. Это поиск целой фразы. Например, если пользователь ввёл `бойцовский клуб`, точное совпадение фразы в названии должно быть очень сильным сигналом.

Именно здесь появляется отличие от простого поиска. Elasticsearch не просто спрашивает “где есть слова?”, он считает, какой результат лучше.

## Слайд 9. Индекс и другая модель поиска

Здесь важно понять разницу в модели.

Простой поиск через `LIKE` можно представить так: база идёт по строкам и проверяет каждую строку. Есть ли там нужный текст? Есть ли там слово? Подходит или нет?

Elasticsearch работает через inverted index. По-русски можно сказать “обратный индекс”.

Обычная таблица устроена так:

```text
фильм -> текст
```

А обратный индекс устроен наоборот:

```text
слово -> список фильмов
```

Например:

```text
тюрьма -> фильм 1, фильм 2, фильм 8
война -> фильм 3, фильм 5
король -> фильм 7
```

Когда пользователь ищет слово, Elasticsearch не обязан думать только как таблица. Он смотрит, какие документы связаны с этим термином, а потом считает их релевантность.

В этом и сила поискового индекса.

## Слайд 10. Цена Elasticsearch

Но Elasticsearch не бесплатный в смысле сложности.

С простым поиском всё просто: данные лежат в PostgreSQL, мы прямо там ищем.

С Elasticsearch появляется отдельный индекс. Это копия части данных, подготовленная специально для поиска.

А раз есть копия, появляется вопрос синхронизации. Что если фильм обновился в PostgreSQL, но индекс ещё не обновился? Что если Elasticsearch недоступен? Что если нужно поменять структуру документа?

В демо это решается просто: есть endpoint `/reindex`, который заново переносит фильмы из PostgreSQL в Elasticsearch.

В реальном проекте это сложнее. Нужно думать про обновления, ошибки, повторные попытки, мониторинг, миграции индекса.

Поэтому Elasticsearch даёт лучший поиск, но добавляет эксплуатационную ответственность.

## Слайд 11. Сравнение двух подходов

Теперь сравним прямо.

SQL `LIKE` — это простая модель: строка содержит подстроку. Например, есть ли внутри описания слово `война`.

Elasticsearch — это модель документов и score. Документ получает оценку: насколько он подходит под запрос.

По языку `LIKE` почти ничего не понимает. Он может привести текст к нижнему регистру, но не понимает морфологию.

Elasticsearch использует analyzer: tokenizer, lowercase, stemmer.

По ранжированию simple search в нашем коде сортирует по рейтингу фильма. Elasticsearch учитывает веса полей и фразовые совпадения.

По данным simple search работает с одной таблицей. Elasticsearch требует отдельный индекс и потом добор полной карточки фильма из базы.

Вывод такой: simple search проще. Elasticsearch мощнее, но сложнее.

## Слайд 12. Как показать это на демо

Если показывать приложение вживую, лучше не просто сказать “вот поиск работает”.

Нужно подобрать запросы, которые показывают разные особенности.

Например, запрос `бойцовский клуб` хорошо показывает точное совпадение фразы в названии фильма.

Запрос `тюрьма` показывает, что поиск может работать не только по названию, но и по описанию фильма.

Запрос `заключённые` подходит для разговора про русский язык и разные формы слов.

Запрос `побег` показывает обычное совпадение по части названия, например с фильмом `Побег из Шоушенка`.

Хороший сценарий такой: сначала показать простой поиск как baseline, то есть базовый вариант. Потом объяснить, что Elasticsearch добавляет analyzer и scoring. Тогда аудитории будет понятно, откуда берётся разница.

## Слайд 13. Как выбрать подход

Самая частая ошибка — выбирать Elasticsearch просто потому, что он звучит мощно.

На самом деле решение зависит от ожиданий пользователя.

Если нужно искать по точному полю, например по id, статусу или категории, Elasticsearch не нужен.

Если нужно просто найти подстроку в маленьком списке, SQL `LIKE` может быть достаточно.

Если появляются язык, формы слов, опечатки, разные веса полей, фразовый поиск и релевантность, тогда уже стоит смотреть в сторону Elasticsearch.

А если поиск становится важной частью продукта и растёт нагрузка, поисковый движок становится ещё полезнее.

Коротко: порог перехода обычно там, где “найти подстроку” превращается в “найти лучший ответ”.

## Слайд 14. Итог

Главный вывод: Elasticsearch нужен не вместо базы данных, а рядом с ней.

PostgreSQL хранит основную правду. Это источник данных.

Elasticsearch отвечает за поиск. Он помогает понять запрос пользователя, обработать текст, посчитать релевантность и вернуть результаты в хорошем порядке.

Обычный поиск спрашивает:

```text
есть ли совпадение в строке?
```

Elasticsearch спрашивает:

```text
какой документ лучше всего подходит под запрос?
```

Но за эту силу мы платим сложностью: нужен индекс, синхронизация и эксплуатация.

Поэтому финальная мысль такая: если нужен простой фильтр — используем обычный поиск. Если нужен настоящий пользовательский поиск с языком и релевантностью — добавляем Elasticsearch.
