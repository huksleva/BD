# Связи между отношениями

## Связь «КЛИЕНТ — ЗАКАЗ»

- Один клиент может иметь много заказов.
    
- Один заказ принадлежит одному клиенту.
    

### Тип связи

- Один-ко-многим
    

### Внешний ключ

В отношение «ЗАКАЗ» добавляется:

- Ид_клиента — FK
    

---

## Связь «СОТРУДНИК — ЗАКАЗ»

- Один сотрудник может вести много заказов.
    
- Один заказ обрабатывается одним сотрудником.
    

### Тип связи

- Один-ко-многим
    

### Внешний ключ

В отношение «ЗАКАЗ» добавляется:

- Таб. номер — FK
    

---

## Связь «ЗАКАЗ — ИГРА»

- Один заказ может содержать много игр.
    
- Одна игра может входить в разные заказы.
    

### Тип связи

- Многие-ко-многим
    

Для реализации связи создается промежуточное отношение.

---

# Отношение «СОСТАВ_ЗАКАЗА»

|Атрибут|Описание|
|---|---|
|Номер|Номер заказа|
|Название|Название игры|
|Производитель|Производитель|
|Количество|Количество экземпляров|

### Потенциальные ключи

- (Номер, Название, Производитель) — PK
    

### Внешние ключи

- Номер — FK
    
- (Название, Производитель) — FK
    

### Обязательные атрибуты

- Количество — NOT NULL
    

### Проверка атрибутов

- Количество > 0
    

---

## Проверка новых атрибутов

После добавления внешних ключей:

- Ид_клиента — NOT NULL
    
- Таб. номер — NOT NULL

## Ответ на вопрос

Если связь между отношениями «ЗАКАЗ» и «СОТРУДНИК» станет один-к-одному, необходимо добавить ограничение UNIQUE для внешнего ключа «Таб. номер» в таблице «ЗАКАЗ».

Это ограничение не позволит одному сотруднику быть ответственным сразу за несколько разных заказов.

## Понятие избыточности данных

Избыточность базы данных — это повторение одинаковых данных в нескольких записях таблиц. Избыточность увеличивает объем хранимой информации и может привести к ошибкам при изменении данных.

---

## Пример неверного логического проектирования

### Таблица заказов

|Номер заказа|Клиент|Телефон|Игра|Производитель|Количество|
|---|---|---|---|---|---|
|1|Петров|8-962-587-33-01|Диксит|Libellud|2|
|1|Петров|8-962-587-33-01|Jungle|Asmodee|1|

### Недостатки

- Повторяются данные клиента;
    
- Дублируется номер телефона;
    
- Повторяется номер заказа;
    
- Усложняется обновление информации.
    

---

## Возможные проблемы

### Аномалия обновления

При изменении телефона клиента необходимо изменять несколько записей.

### Аномалия удаления

При удалении последнего заказа можно потерять информацию о клиенте.

### Аномалия добавления

Нельзя добавить нового клиента без создания заказа.

---

## Способы устранения избыточности

### 1. Нормализация данных

Разделение информации на несколько таблиц.

### 2. Выделение сущностей

Создание отдельных таблиц:

- КЛИЕНТ
    
- ЗАКАЗ
    
- ИГРА
    
- СОСТАВ_ЗАКАЗА
    

### 3. Использование ключей

- Первичные ключи для идентификации записей;
    
- Внешние ключи для связи таблиц.
    

### 4. Исключение повторяющихся данных

Информация о клиенте хранится отдельно и не дублируется в заказах.
